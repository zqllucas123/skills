# SWE.1 软件需求分析 - 详细工作流指南

## 过程目的

SWE.1 的目的是建立一个结构化的分析，将系统需求（或用户需求）转化为软件需求。软件需求应清晰、无歧义、可验证，并建立与上层需求的追溯关系。

## 输入要求

| 输入项 | 来源 | 说明 |
|--------|------|------|
| 系统需求 | SYS.2 | 系统层面的功能和非功能需求 |
| 系统架构 | SYS.3 | 系统组件分配和接口定义 |
| 用户需求 | 用户/客户 | PRD、功能描述或直接输入 |
| 利益相关者需求 | 各相关方 | 质量属性、约束条件等 |

> **注意**：若未提供系统需求和系统架构，则直接将用户输入作为软件需求的源头。

## 执行步骤

### Step 1: 需求收集与分析

1. **读取输入文档**
   - 使用文件读取工具获取文档内容
   - 若为 PDF/Word 格式，提取文本内容
   - 若为自然语言描述，直接解析

2. **识别需求类型**
   - 功能性需求：描述系统应该做什么
   - 非功能性需求：性能、安全、可靠性等
   - 约束需求：技术限制、法规要求等

3. **消除歧义**
   - 识别模糊表述并澄清
   - 统一术语定义
   - 确保需求可理解和可测试

### Step 2: 需求结构化

1. **分配唯一标识符**
   ```
   格式：SWR-XXX
   示例：SWR-001, SWR-002, ...
   ```

2. **需求属性定义**
   | 属性 | 说明 | 示例值 |
   |------|------|--------|
   | ID | 唯一标识符 | SWR-001 |
   | 标题 | 需求简述 | 用户登录功能 |
   | 描述 | 详细描述 | 系统应支持用户通过账号密码登录 |
   | 优先级 | 重要程度 | 高/中/低 或 1-5 |
   | 类型 | 需求类型 | 功能/非功能/约束 |
   | 状态 | 当前状态 | 草稿/评审/批准 |
   | 验收标准 | 验证条件 | 给定条件下的预期结果 |
   | 来源 | 需求来源 | UR-001 或 PRD章节 |

3. **需求分解原则**
   - 每条需求应该是原子性的
   - 避免复合需求（如"系统应该A和B"）
   - 每条需求应该是可独立验证的

### Step 4: 建立追溯性

**产出文件**: `13-51_Traceability_Matrix.csv`

**初始结构**:
```csv
User_Req_ID,SW_Req_ID,Relationship,Notes
UR-001,SWR-001,satisfies,功能需求分解
UR-001,SWR-002,satisfies,功能需求分解
UR-002,SWR-003,satisfies,性能需求转化
```

**追溯性规则**:
- 每条软件需求必须追溯至少一条用户/系统需求
- 关系类型：satisfies（满足）、derives（派生）、refines（细化）
- 双向追溯：可以从用户需求查到软件需求，也可以从软件需求查到用户需求

### Step 5: 自检验证

**检查清单**:
- [ ] 所有用户需求是否都已覆盖？
- [ ] 每条软件需求是否清晰无歧义？
- [ ] 需求是否可验证（有明确的验收标准）？
- [ ] 需求标识符是否唯一且符合规范？
- [ ] 可追溯性矩阵是否完整？
- [ ] 需求之间是否存在冲突？

## 最佳实践

1. **需求完整性检查**
   - 使用 INVEST 原则验证需求质量
   - Independent（独立）、Negotiable（可协商）、Valuable（有价值）、Estimable（可估算）、Small（小型）、Testable（可测试）

2. **需求优先级排序**
   - MoSCoW 方法：Must have、Should have、Could have、Won't have
   - 或使用数值评分：1-5 或 1-10

3. **需求评审**
   - 在进入下一阶段前，建议进行需求评审
   - 检查需求的一致性、完整性、可行性

## 常见问题与解决

| 问题 | 解决方案 |
|------|----------|
| 需求模糊 | 通过与用户沟通澄清，或做出合理假设并记录 |
| 需求冲突 | 识别冲突点，根据优先级解决 |
| 需求遗漏 | 对照用户需求逐项检查，补充遗漏项 |
| 需求过大 | 分解为多个原子需求 |
